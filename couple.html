The Three "WHY"s of Couple

Understanding the design philosophies that drive our privacy architecture and market potential.

01

Why Anonymity?

By rejecting emails, phone numbers, and social logins, Couple eliminates the vulnerability vectors of SIM-swapping, metadata triangulation, and server leaks. A 16-word local phrase hashed via Argon2id ensures users have total cryptographic ownership over their credentials without leaving a trace.

02

Why Relationship Lock?

Couple rejects the social graph. By locking two profiles together under a single active ID at the database layer, it blocks spam, harassment, and third-party contact lists. It stops being a standard utility tool and instead becomes a dedicated virtual room reserved entirely for one relationship.

03

Why Auto-Deletion?

Absolute digital privacy requires a finite data lifecycle. Auto-deleting accounts inactive for 90 days prevents stale metadata reservoirs from existing indefinitely. If a couple steps away, their entire shared room, E2EE keys, and history dissolve automatically, leaving a zero-footprint trace.

The A-to-R Features Matrix

Filter or search features dynamically to understand the full architectural scope of Couple.

A. Anonymous Sign-Up

Register with only Username + Password. Avoids phone, email, and social login hooks to prevent location metadata harvesting.

B. 16-Word Recovery Seed

Generates a bip39-standard recovery seed displayed only once on account registration. Essential for offline backups.

C. Argon2id Hash Storage

Plaintext seeds are never stored or transmitted. The server saves only the Argon2id key-derivative hash for recovery matching.

D. Couple Code Connection

Pairs partners via an ephemeral 6-digit verification code. Bypasses public indexing, search inputs, or username queries.

E. Relationship Lock

Restricts conversation participation using PostgreSQL database constraints. Enforces a maximum participant count of 2 per user.

F. Single Chat Workspace

Removes navigation boards, chat list panels, and contacts grids. Authenticating leads directly into the active conversation.

G. E2EE Double Ratchet

Client-side cryptographic keys derived from an ephemeral X3DH handshake ensure all messages are private before transit.

H. Presence & Typing

Monitors online connectivity states and typing indicators in real time using Supabase Realtime Broadcast pipelines.

I. Ticks & Read Status

M3 style ticks: 1 tick represents sent status, 2 ticks delivered status, and colored ticks read status.

J. Asymmetric Corners

Custom M3 bubble design where incoming bubbles have a sharp bottom-left corner and outgoing bubbles have a sharp bottom-right corner.

K. Swipe Actions

Drag messages with haptic interpolation triggers to access quick reply configurations.

L. Long-Press Bottom Sheet

Triggers Copy, Edit, Reply, and Delete actions underneath a dynamic Material You reactions sheet.

M. Material Motion

Every dynamic element leverages custom 300ms spring physics animations and ink-ripple overlay feedback.

N. Client-Encrypted Media

Uploads chat media to ImgBB via API, but stores links within client-encrypted payloads. The server only sees ciphertext links.

O. Device Session List

Displays current active sessions containing country, browser name, and last active timestamp. Excludes IP tracking.

P. 90-Day Auto-Deletion

Monitors login activity. Inactivity past 90 consecutive days triggers a complete account/message cascade wipe.

Q. Couple Dissolution Flow

Disconnecting requires dual confirmation and enters a 24-hour cooling-off state before unlinking.

R. Dynamic Adaptive Palette

Responsive Flutter layout dynamic color system adapting to system wallpaper accents automatically.
